# Root Cause Analysis (BABOK 10.40)

## Overview

Root Cause Analysis (RCA) is a systematic problem-solving technique used to identify the underlying origins of an operational problem, system failure, or process bottleneck. Instead of merely addressing visible symptoms, RCA aims to find and mitigate the systemic root causes (often related to People, Process, Technology, or Environment) to prevent recurrence. Key frameworks include **The 5 Whys** and **Ishikawa (Fishbone) Diagrams**.

## When to Use

- User needs to diagnose why a system outage occurred, why a feature failed QA, or why a business KPI dropped.
- User needs to recommend corrective actions for process bottlenecks or quality defects.
- Task 3.4 (Plan Business Analysis Information Management), Task 6.1 (Analyze Current State), Task 6.3 (Assess Risks), Task 8.3 (Assess Solution Limitations).

## Do not use when
- Brainstorming new system features or forward-looking business strategies (use User Stories 10.48 or Use Cases 10.47 instead).
- Designing database models or interfaces (use Data Dictionary 10.12 or Interface Analysis 10.24 instead).

## Quick Reference

| RCA Framework | Focus Area | When to Use |
| :--- | :--- | :--- |
| **5 Whys Analysis** | Deep-dive vertical tracing | For simple to moderately complex causal chains |
| **Ishikawa (Fishbone)** | Broad categorization (6Ms/4Ps) | For complex failures with multiple contributing factors |
| **Primary KAs** | KA03, KA06, KA07, KA08 | |
| **Most common tasks** | Task 6.1 (Current State), Task 8.3 (Assess Limitations) | |

## Step-by-Step for AI Agent

1. **Verify Context Inputs:** Identify the primary symptom or problem statement (e.g., "Customer checkouts failing at payment step").
2. **⚠️ SHARP NEGATIVE CONSTRAINT:** Every root cause analysis must end in an actionable systemic mitigation. If the user describes a problem but refuses to provide baseline system behaviors, logs, or operational facts, STOP. You must not invent fictitious system errors (like "corrupt hard drive") without input data. Ask the user for specific error logs, timing details, or process behaviors.
3. **Execute 5 Whys Analysis:** Formulate a logical, vertical causal chain. Each "Why" must naturally flow from the previous step. Stop when a systemic, fixable root cause is identified.
4. **Construct Ishikawa (Fishbone) Model:** Group contributing factors into standard categories: *Technology/Systems, Process/Methods, People/Roles, and Environment/Policies*.
5. **Formulate Corrective Actions:** Define immediate hotfixes (short-term) and systemic mitigations (long-term).
6. **Generate Output:** Populate the standard template below.

## Common Mistakes & CBAP Gotchas

> **⚠️ AI AGENT INSTRUCTION:** If any of these traps are detected in the user's request, surface a `⚠️ CBAP Gotcha` alert before generating the output.

**Trap 1 — Blaming Human Error as a Root Cause**
- Wrong: Concluding that the root cause is *"Operator fell asleep"* or *"Developer made a typo"*.
- Right: Human error is a symptom. The systemic root cause is the lack of guardrails, code reviews, automated tests, or recovery procedures that allowed the human error to occur. Always dig deeper into system designs.

**Trap 2 — Lack of Causal Linking**
- Wrong: Jumping directly from Symptom: *"Failed login"* to Root Cause: *"AWS went down"* without verifying intermediate steps.
- Right: Ensure every step of the 5 Whys is logical, proven by logs, and connected.
