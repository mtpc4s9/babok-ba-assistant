# Data Flow Diagrams (BABOK 10.13)

## Overview

A Data Flow Diagram (DFD) visually represents how data moves through an information system. It outlines the sources/destinations of data (External Entities), the transformations performed (Processes), the storage points (Data Stores), and the transmission paths (Data Flows). DFDs are structured hierarchically:
- **Context Level (Level 0 DFD):** High-level view showing system boundaries and external entity relationships.
- **Level 1 DFD:** Decomposes the main processes of the system into finer functional steps, revealing internal data stores.

## When to Use

- User needs to capture how data inputs from external stakeholders are processed and stored.
- User needs to establish clear architectural boundaries of a solution.
- Task 7.1 (Specify and Model Requirements), Task 7.4 (Define Requirements Architecture).

## Do not use when
- Modeling complex control logic, sequence timings, or conditional looping paths (use Process Modelling 10.35 or Sequence Diagrams instead).
- Designing the physical storage relationship of database tables (use Data Dictionary 10.12 or Data Modelling instead).

## Quick Reference

| DFD Element | Standard Representation | What it represents |
| :--- | :--- | :--- |
| **External Entity** | Square / Rectangle | Sources or sinks of data outside system control |
| **Process** | Rounded Rectangle / Circle | Data transformers (must have input AND output) |
| **Data Store** | Open-ended Rectangle | Databases, logs, or file storage points |
| **Data Flow** | Directed Arrow | Path of data movement (labeled with data payload) |
| **Primary KAs** | KA05, KA07 | |
| **Most common tasks** | Task 7.1 (Specify & Model), Task 7.4 (Architecture) | |

## Step-by-Step for AI Agent

1. **Verify Context Inputs:** Scan for system actors, databases, and key data ingestion/processing steps.
2. **⚠️ SHARP NEGATIVE CONSTRAINT:** A DFD processes data, it does not represent business procedural steps. If the user describes actions with no data outputs (e.g., "User waits for approval"), STOP. Request clarification on what data is created or modified. Every process in a DFD must have at least one input and one output flow. Avoid "black holes" (inputs only) or "miracles" (outputs only).
3. **Draft Context Level (Level 0):** Map the system as a single central process node and link it to external entities.
4. **Draft Level 1 Decomposition:** Break the system node into 3 to 7 core processes, showing where data is written to or read from internal Data Stores.
5. **Generate Output:** Render DFD structures using high-fidelity Mermaid diagrams and tabular catalogs.

## Template: Data Flow Diagram Spec

### 1. DFD Level 0 (Context Level Diagram)

```mermaid
graph TD
    %% Context Level DFD
    subgraph System Boundary
        SYS["1.0 Core System Process"]
    end
    
    %% External Entities
    EE1["Customer (Actor)"]
    EE2["Payment Gateway (External System)"]
    
    %% Data Flows
    EE1 -->| "Login Credentials & Order Data" | SYS
    SYS -->| "Order Confirmation & Invoice" | EE1
    SYS -->| "Payment Processing Request" | EE2
    EE2 -->| "Payment Authorization Response" | SYS
```

### 2. DFD Level 1 (Process Decomposition)

```mermaid
graph TD
    %% Level 1 DFD
    subgraph Processes
        P1["1.1 Authenticate User"]
        P2["1.2 Process Order"]
        P3["1.3 Authorize Payment"]
    end
    
    subgraph Data Stores
        DS1[("DS-1: User DB")]
        DS2[("DS-2: Order DB")]
    end
    
    %% External Entities
    Customer["Customer (Actor)"]
    Gateway["Payment Gateway"]
    
    %% Flows
    Customer -->| "Login Inputs" | P1
    P1 -->| "User Query" | DS1
    DS1 -->| "User Record" | P1
    P1 -->| "Auth Token" | Customer
    
    Customer -->| "Order Payload" | P2
    P2 -->| "Write Order" | DS2
    P2 -->| "Tx Payload" | P3
    
    P3 -->| "API Request" | Gateway
    Gateway -->| "API Response" | P3
    P3 -->| "Update Payment Status" | DS2
    P2 -->| "Invoice Data" | Customer
```

## Common Mistakes & CBAP Gotchas

> **⚠️ AI AGENT INSTRUCTION:** If any of these traps are detected in the user's request, surface a `⚠️ CBAP Gotcha` alert before generating the output.

**Trap 1 — Miracle Processes (Output without Input)**
- Wrong: A process node labeled *"Generate Invoices"* that has outgoing arrows to Customer, but no incoming arrows from databases or order flows.
- Right: Connect *"Generate Invoices"* to both an incoming *"Read Order"* flow from the Order Database and a triggering event.

**Trap 2 — Data Store to Data Store Flows**
- Wrong: Drawing a direct data flow arrow from one database symbol (`DS-1`) directly to another (`DS-2`).
- Right: Data stores cannot communicate directly; they must always pass through a Process node that reads from one store and writes to the other.
