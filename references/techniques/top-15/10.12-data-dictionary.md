# Data Dictionary (BABOK 10.12)

## Overview

A Data Dictionary (or metadata registry) standardizes the definitions, data structures, and relationships of data elements in an enterprise or software system. It ensures that all stakeholders (business, analysts, and developers) interpret data fields consistently, preventing translation errors during system design and integration.

## When to Use

- User needs to specify the exact schema requirements for a new feature, database table, or API contract.
- User needs to translate business concepts into precise physical data representations.
- Task 7.1 (Specify and Model Requirements), Task 7.4 (Define Requirements Architecture).

## Do not use when
- Designing high-level entity-relationship diagrams (use Data Modelling 10.15 instead).
- Documenting operational policies and declarative constraints (use Business Rules Analysis 10.09 instead).

## Quick Reference

| Field Element | Mandatory Parameters | Why it matters |
| :--- | :--- | :--- |

| **Alias** | Business vs. Technical names | Resolves naming misalignment |

## Step-by-Step for AI Agent

1. **Verify Context Inputs:** Scan for list of data fields, business requirements, or raw spreadsheet templates provided by the user.
2. **⚠️ SHARP NEGATIVE CONSTRAINT:** If the data types or formatting rules of fields are completely unknown, STOP. Request the user to provide the core business rules or sample records. Do not guess database types or validation formats without baseline logic.
3. **Establish Names & Aliases:** Define the standard technical name (snake_case/camelCase) and map it to the business term.
4. **Specify Domain & Bounds:** Define the validation logic, patterns (e.g., regex for emails), nullable status, and default values.
5. **Generate Output:** Populate the standard template below.

## Common Mistakes & CBAP Gotchas

> **⚠️ AI AGENT INSTRUCTION:** If any of these traps are detected in the user's request, surface a `⚠️ CBAP Gotcha` alert before generating the output.

**Trap 1 — Inconsistent Naming Conventions**
- Wrong: Mixing `CustomerId` (PascalCase), `order_id` (snake_case), and `ItemNo` in the same schema specifications.
- Right: Maintain rigid consistent naming (e.g., all fields in snake_case or camelCase matching backend rules).

**Trap 2 — Weak Data Domain Specifications**
- Wrong: Defining a status domain as *"depends on the user status"* or *"any text"*.
- Right: Enforce strict enum lists: `['ACTIVE', 'INACTIVE']` or point to a master table constraint.
