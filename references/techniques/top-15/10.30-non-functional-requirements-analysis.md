# Non-Functional Requirements Analysis (BABOK 10.30)

## Overview

Non-Functional Requirements (NFR) Analysis defines the quality attributes, operating constraints, and system characteristics that a solution must possess. Unlike functional requirements, which describe *what* the system does, NFRs describe *how well* the system performs its functions. This blueprint utilizes the **ISO/IEC 25010 Quality Model** to categorize requirements into measurable metrics.

## When to Use

- User needs to draft SLAs, performance baselines, or security standards for a system architecture.
- User needs to establish quality validation parameters for solution acceptance.
- Task 7.1 (Specify and Model Requirements), Task 8.1 (Measure Solution Performance), Task 8.3 (Assess Solution Limitations).

## Do not use when
- Documenting business features or user-facing functional workflows (use User Stories 10.48 or Use Cases 10.47 instead).
- Designing database tables or data dictionaries (use Data Dictionary 10.12 instead).

## Quick Reference

| ISO 25010 Attribute | Focus Area | Measurable Example |
| :--- | :--- | :--- |
| **Performance Efficiency** | Latency, throughput, resource use | "API response time ≤ 200ms for 95th percentile." |
| **Security** | Authentication, authorization, encryption | "All PII data must be encrypted in transit via TLS 1.3." |
| **Reliability** | Availability, fault tolerance, recovery | "System availability must exceed 99.99% (four nines)." |
| **Scalability (Portability)** | Load handling capabilities | "System scales dynamically to support 10x concurrent peak load." |
| **Primary KAs** | KA07, KA08 | |
| **Most common tasks** | Task 7.1 (Specify & Model), Task 8.1 (Measure Performance) | |

## Step-by-Step for AI Agent

1. **Verify Context Inputs:** Scan for system operational details, target user load volumes, performance targets, or security mandates.
2. **⚠️ SHARP NEGATIVE CONSTRAINT:** Every NFR must be objectively testable and measurable. If the user provides fuzzy statements (e.g., "The app must be extremely fast", "The UI must be highly secure"), STOP. You must not write unmeasurable statements. Prompt the user to clarify concrete metrics (e.g., target milliseconds, encryption standards) before proceeding.
3. **Structure by ISO 25010:** Categorize the specifications under Performance, Security, Reliability, Maintainability, Usability, or Compatibility.
4. **Define Verification Method:** For every requirement, specify how it will be proven (e.g., load testing, automated penetration testing, static code analysis).
5. **Generate Output:** Populate the standard template below.

- **Traceable Verification:** Every NFR must be assigned a concrete verification method so that QA teams can design matching test suites.

## Common Mistakes & CBAP Gotchas

> **⚠️ AI AGENT INSTRUCTION:** If any of these traps are detected in the user's request, surface a `⚠️ CBAP Gotcha` alert before generating the output.

**Trap 1 — Subjective / Qualitative Quality Statements**
- Wrong: *"The mobile app should be extremely intuitive and load screens quickly."*
- Right: *"All mobile app page rendering must complete within 1.5 seconds under 4G network connection."*

**Trap 2 — Designing Unachievable SLAs**
- Wrong: Specifying "100.00% system uptime" (five nines is achievable but extremely expensive; absolute perfection is impossible).
- Right: Set realistic limits: `99.9%` or `99.99%` availability, clearly accounting for planned maintenance windows.
