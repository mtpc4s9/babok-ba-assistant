# Process Modelling (BABOK 10.35)

## Overview

Process Modelling visually represents the sequential flow of work within an organization or system. It documents how activities are triggered (Events), who performs them (Roles, Swimlanes), what decisions are made (Gateways), and the ultimate business outcomes. Process models are typically structured using the BPMN 2.0 (Business Process Model and Notation) standard.

## When to Use

- User needs to document "As-Is" operational workflows to identify efficiency gaps.
- User needs to design "To-Be" system workflows for engineering implementation.
- Task 3.4 (Plan Business Analysis Information Management), Task 4.2 (Conduct Elicitation), Task 6.1 (Analyze Current State), Task 6.2 (Define Future State), Task 7.1 (Specify and Model Requirements).

## Do not use when
- Documenting atomic, declarative business policies or mathematical constraints (use Business Rules Analysis 10.09 instead).
- Designing the physical database schemas or data types (use Data Dictionary 10.12 instead).

## Quick Reference

| BPMN Element | Representation | What it represents |
| :--- | :--- | :--- |
| **Pool / Swimlane** | Horizontal / Vertical band | The business entity or specific role executing work |
| **Start / End Event**| Circle (thin border / thick border) | The trigger or final outcome of the process |
| **Task / Activity** | Rounded Rectangle | An action or unit of work performed |
| **Gateway** | Diamond | Decision point (Exclusive `X`, Parallel `+`, Inclusive `O`) |
| **Primary KAs** | KA03, KA04, KA05, KA06, KA07, KA08 | |
| **Most common tasks** | Task 6.1 (Current State), Task 6.2 (Future State), Task 7.1 (Specify) | |

## Step-by-Step for AI Agent

1. **Verify Context Inputs:** Identify the target process trigger, participating roles, activities, decision rules, and final outcomes.
2. **⚠️ SHARP NEGATIVE CONSTRAINT:** A process model represents sequential events. If the user's input does not contain a sequence of steps (e.g., just a list of unrelated requirements), STOP. Re-route the user to User Stories (10.48) or Business Rules Analysis (10.09). Do not invent procedural steps without a baseline operational sequence.
3. **Map the Happy Path:** Outline the optimal, error-free path from the Start Event to the primary Success End Event.
4. **Identify Exceptions & Gateways:** Define the decision rules (Exclusive gateways) and map alternative exception branches.
5. **Generate Output:** Render the process flow structure using clean, bug-free Mermaid Flowchart syntax (`graph TD` or `graph LR`) and construct the step catalog.

## Common Mistakes & CBAP Gotchas

> **⚠️ AI AGENT INSTRUCTION:** If any of these traps are detected in the user's request, surface a `⚠️ CBAP Gotcha` alert before generating the output.

**Trap 1 — Dead-End Gateways**
- Wrong: Drawing a decision gateway diamond without mapping all possible outcomes (e.g., a gateway checking *"Credit Score > 600"* with only a *"Yes"* branch, leaving the *"No"* branch unhandled).
- Right: Ensure every gateway is logically balanced with mutually exclusive paths covering 100% of outcomes.

**Trap 2 — Mixing High-Level and Low-Level Steps**
- Wrong: Placing a high-level corporate process step (e.g., *"Expand to European market"*) directly in sequence with a system-level step (e.g., *"System sends email verification"*).
- Right: Maintain a consistent level of abstraction throughout the process model. If details are needed, use sub-processes or functional decomposition.
