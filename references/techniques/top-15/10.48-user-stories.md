# User Stories (BABOK 10.48)

## Overview

A User Story is a high-level, customer-centric statement of functionality used in Agile frameworks to represent backlog items. It captures what a user needs to achieve and why, without specifying the technical implementation details. A high-quality user story consists of:
- **The Card (Narrative):** Follows the standard `As a... I want to... So that...` structure.
- **Acceptance Criteria (Validation):** Formatted using **Gherkin syntax** (`Given-When-Then`) to define the exact boundaries of the story and guide QA test designs.

## When to Use

- User needs to draft an Agile product backlog for software sprint planning.
- User needs to translate technical capabilities into user-centric value blocks.
- Task 5.1 (Prepare for Elicitation), Task 7.1 (Specify and Model Requirements), Task 7.4 (Define Requirements Architecture).

## Do not use when
- Documenting highly technical, system-to-system integrations with complex data payloads (use Interface Analysis 10.24 instead).
- Designing database models or metadata standards (use Data Dictionary 10.12 instead).

## Quick Reference

| Story Component | Syntax Structure | What it represents |
| :--- | :--- | :--- |
| **As a [Role]** | Target persona / user category | Who receives the value |
| **I want to [Action]**| Target action / system capability | What the system must allow |
| **So that [Benefit]** | Core business value achieved | Why this action matters |
| **Given [Context]** | Initial state baseline | Prerequisites for test case |
| **When [Event]** | Trigger event action | The action taken by the user |
| **Then [Outcome]** | Expected system result | Testable pass/fail outcome |
| **Primary KAs** | KA05, KA07 | |
| **Most common tasks** | Task 7.1 (Specify & Model), Task 7.4 (Architecture) | |

## Step-by-Step for AI Agent

1. **Verify Context Inputs:** Scan for target user personas, system workflows, or high-level business goals.
2. **⚠️ SHARP NEGATIVE CONSTRAINT:** A user story represents a slice of system value. If the user provides high-level business goals with no system touchpoints (e.g., "We want to expand our business to Europe"), STOP. Re-route the user to Business Cases (10.07). Do not write stories with no system-actor boundaries.
3. **Draft the Narrative Card:** Follow the strict three-part Agile structure. Ensure the "So that" specifies a *business or customer benefit*, not just a repeat of the action.
4. **Enforce INVEST Criteria:**
   - **I**ndependent, **N**egotiable, **V**aluable, **E**stimable, **S**mall, **T**estable.
5. **Construct Gherkin Acceptance Criteria:** Write at least two distinct `Given-When-Then` scenarios (one positive/happy path, one negative/error path).
6. **Generate Output:** Populate the standard template below.

## 🤖 AI AGENT INSTRUCTION (EXECUTION RULES)
- **Zero Technical How-To:** Never describe database tables, specific API paths, or code logic inside the User Story Narrative or Acceptance Criteria. Keep them strictly functional.
- **Value-Driven 'So That':** Ensure the "So that..." part of the story states a clear benefit (e.g., "So that I can complete checkouts without re-entering my payment details" rather than "So that I can save my card").

## Template: Agile User Story Specification

### 1. User Story Profile
- **Story ID:** `US-101`
- **Feature Area:** [INSERT: Name, e.g., "Customer Checkout & Payments"]
- **Assigned Epic:** [INSERT: Epic Name, e.g., "Online Payment Integration"]
- **Estimated Effort:** [INSERT: Story Points | e.g., "3 Story Points"]

### 2. The User Story Card (Narrative)
- **AS A:** [INSERT: Target Persona/Role, e.g., "Registered E-Commerce Customer"]
- **I WANT TO:** [INSERT: Desired System Action, e.g., "securely save my credit card details in my account profile"]
- **SO THAT:** [INSERT: Core Value/Benefit, e.g., "I can complete future checkouts faster without re-entering my card number"]

### 3. Acceptance Criteria (Gherkin Format)

#### Scenario 1: Successfully saving a valid credit card (Happy Path)
- **GIVEN:** [INSERT: Context, e.g., "the Customer is on the 'Payment Methods' settings page"]
- **AND:** [INSERT: Context, e.g., "the Customer has no previously saved payment cards"]
- **WHEN:** [INSERT: Event, e.g., "the Customer enters a valid credit card number, expiration date, CVV, and clicks 'Save Card'"]
- **THEN:** [INSERT: Expected Outcome, e.g., "the System validates the card structure and contacts the gateway for a token"]
- **AND:** [INSERT: Expected Outcome, e.g., "the System saves the secure card token and displays a 'Card Saved Successfully' message"]

#### Scenario 2: Error handling for invalid expiration date (Negative Path)
- **GIVEN:** [INSERT: Context, e.g., "the Customer is on the 'Payment Methods' settings page"]
- **WHEN:** [INSERT: Event, e.g., "the Customer enters a card with an expiration date in the past and clicks 'Save Card'"]
- **THEN:** [INSERT: Expected Outcome, e.g., "the System blocks the submission and highlights the expiration date field in red"]
- **AND:** [INSERT: Expected Outcome, e.g., "the System displays an error message stating: 'Card expiration date must be in the future.'"]

## Common Mistakes & CBAP Gotchas

> **⚠️ AI AGENT INSTRUCTION:** If any of these traps are detected in the user's request, surface a `⚠️ CBAP Gotcha` alert before generating the output.

**Trap 1 — Writing User Stories as Developer Tasks**
- Wrong: *"As a developer, I want to create a database table named 'card_tokens' so that user cards can be stored."*
- Right: *"As a customer, I want to save my payment details so that I don't have to enter them every time."* (Keep stories focused on user value).

**Trap 2 — Weak and Vague Acceptance Criteria**
- Wrong: *"Verify that the save card button works and has no bugs."* (Completely unmeasurable for QA).
- Right: Use strict Gherkin syntax to map exact pre-conditions, events, and validation outcomes.
