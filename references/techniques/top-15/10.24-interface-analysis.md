# Interface Analysis (BABOK 10.24)

## Overview

Interface Analysis systematically identifies and describes the interaction boundaries between systems, hardware devices, external actors, or business roles. It establishes what information is exchanged (data payload), how it is exchanged (protocol, frequency, transport mechanism), and the validation constraints (error handling, security) governing the connection.

## When to Use

- User needs to design integration requirements for a third-party service (e.g., payment gateway, SMS service).
- User needs to specify internal component boundaries (microservices communications, frontend-to-backend API contracts).
- Task 4.2 (Conduct Elicitation), Task 5.3 (Assess Requirements Changes), Task 7.1 (Specify and Model Requirements).

## Do not use when
- Documenting general business rules independent of system communication (use Business Rules Analysis 10.09 instead).
- Defining the database storage entity schemas (use Data Dictionary 10.12 instead).

## Step-by-Step for AI Agent

1. **Verify Context Inputs:** Identify the interacting entities (e.g., "Internal CRM" and "SendGrid API") and the transaction data requirements.
2. **⚠️ SHARP NEGATIVE CONSTRAINT:** If the communication direction, payload elements, or authentication methods are completely unspecified, STOP. Request the connection endpoints or API documentation links. Never invent system API endpoint names or security tokens (like `/api/v1/fake-charge`) without a baseline reference.
3. **Map Connection Metadata:** Establish protocols (HTTPS, SFTP, AMQP), format types (JSON, XML, CSV), trigger events, and call frequencies.
4. **Draft Payload Specs:** Model the request and response structures, highlighting key validation constraints and mandatory elements.
5. **Generate Output:** Present the integration specifications using structured markdown tables and JSON payload representations.

## Common Mistakes & CBAP Gotchas

> **⚠️ AI AGENT INSTRUCTION:** If any of these traps are detected in the user's request, surface a `⚠️ CBAP Gotcha` alert before generating the output.

**Trap 1 — One-Way Interface Definition**
- Wrong: Mapped the request payload, but completely omitted the response structures, error codes, and timeout limits.
- Right: System interfaces always have a return behavior. Always specify the response payloads (including positive, negative, and exception handling paths).

**Trap 2 — Weak Validation Definitions**
- Wrong: Stating a payload parameter is *"user details"* or *"customer information"*.
- Right: Define the exact schema: `customer_name` (String, min 2 chars, max 50 chars).
