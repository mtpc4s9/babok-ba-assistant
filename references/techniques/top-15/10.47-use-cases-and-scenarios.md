# Use Cases and Scenarios (BABOK 10.47)

## Overview

Use Cases and Scenarios describe how a specific role or entity (Actor) interacts with a software solution to achieve a defined business goal. Unlike general requirements lists, Use Cases capture system behavior in a structured narrative format, outlining:
- **Pre-conditions:** Criteria that must be true before the Use Case can execute.
- **Post-conditions:** The guaranteed state of the system after execution (both success and failure).
- **Main Flow (Happy Path):** The sequential, error-free path to success.
- **Alternate Flows & Exceptions:** How the system handles deviations and errors.

## When to Use

- User needs to translate business requirements into detailed functional specifications for developers.
- User needs to design end-to-end integration or system validation test suites.
- Task 5.1 (Prepare for Elicitation), Task 7.1 (Specify and Model Requirements), Task 7.4 (Define Requirements Architecture).

## Do not use when
- Documenting agile backlogs with highly granular, user-centric statements (use User Stories 10.48 instead).
- Specifying database tables or data dictionaries (use Data Dictionary 10.12 instead).

## Quick Reference

| Component | Standard Representation | What it represents |
| :--- | :--- | :--- |
| **Actor** | Stick figure / Named Role | User role or external system initiating interaction |
| **Pre-condition** | Testable state statement | Prerequisites (e.g., "User is logged in") |
| **Post-condition (Success)** | Testable system state | Value achieved (e.g., "Transaction recorded in DB") |
| **Alternate Flow** | Branch step notation (e.g., `A1: Step 3`) | Successful detour from the main flow |
| **Exception Flow** | Branch step notation (e.g., `E1: Step 4`) | Failure detour that terminates execution |
| **Primary KAs** | KA05, KA07, KA08 | |
| **Most common tasks** | Task 7.1 (Specify & Model), Task 7.4 (Architecture) | |

## Step-by-Step for AI Agent

1. **Verify Context Inputs:** Identify the Actor, the target System Goal, and the raw requirement details.
2. **⚠️ SHARP NEGATIVE CONSTRAINT:** A Use Case is a detailed system interaction. If the user's input only contains business goals without system details (e.g., "We want to increase sales by 10%"), STOP. Re-route the user to Business Cases (10.07). Do not invent system-level actor steps without operational goals.
3. **Draft Pre-conditions & Post-conditions:** Enforce strict, measurable states (e.g., "Account is active", "Balance is debited").
4. **Draft Main Flow:** Describe a vertical, numbered, alternating narrative sequence (e.g., "1. Actor does X, 2. System does Y"). Keep statements active and concise.
5. **Map Detours (Alternates/Exceptions):** Identify at which steps failures or options occur, and write separate branching paths.
6. **Generate Output:** Populate the standard template below.

## 🤖 AI AGENT INSTRUCTION (EXECUTION RULES)
- **Active Alternating Voice:** Ensure that every step in the flows clearly specifies who executes the action (e.g., "The Actor enters...", "The System validates..."). Avoid passive voice.
- **Fail-State Postconditions:** Always define a separate "Post-condition (Failure)" to specify what happens to system state (e.g., transaction rollback) if an exception flow occurs.

## Template: Use Case Specification

### 1. Use Case Profile
- **Use Case ID:** `UC-101`
- **Use Case Name:** [INSERT: Standard Name, e.g., "Submit Order Checkout"]
- **Primary Actor:** [INSERT: Actor Name, e.g., "Registered Customer"]
- **Brief Description:** [INSERT: Short text | What the use case achieves | Process Model]
- **Pre-conditions:**
  1. [INSERT: State, e.g., "Customer has a valid active account and is logged in."]
  2. [INSERT: State, e.g., "Customer has at least one item in their active shopping cart."]
- **Post-conditions (Success):**
  1. [INSERT: State, e.g., "Order is successfully placed and marked as 'PAID'."]
  2. [INSERT: State, e.g., "Invoice notification is queued for delivery."]
- **Post-conditions (Failure):**
  1. [INSERT: State, e.g., "No funds are debited from the customer's account."]
  2. [INSERT: State, e.g., "Cart contents remain intact and unchanged."]

### 2. Main Flow (Happy Path)

| Step | Action Description (Actor / System alternating) |
| :--- | :--- |
| **Step 1** | **Actor:** The customer clicks the "Proceed to Checkout" button. |
| **Step 2** | **System:** Fetches current cart details and prompts the customer to select shipping address and payment method. |
| **Step 3** | **Actor:** The customer selects a saved shipping address and enters valid credit card details, then clicks "Pay Now". |
| **Step 4** | **System:** Validates credit card format and submits transaction payload to the Payment Gateway (INT-101). |
| **Step 5** | **System:** Debits funds, records order in the Order Database (DS-2), and displays a "Payment Successful" screen. |

### 3. Detours (Alternate & Exception Flows)

#### Alternate Flow A1: Use Saved Payment Method
*Triggered at Step 3 of Main Flow when Customer selects a pre-saved credit card.*
- **A1.1:** The customer selects a saved card token instead of entering new details.
- **A1.2:** System validates token validity. Flow resumes at Step 4.

#### Exception Flow E1: Payment Declined
*Triggered at Step 4 of Main Flow when Payment Gateway returns a decline response.*
- **E1.1:** The system displays an error message stating: "Payment Declined. Please try a different card."
- **E1.2:** System rolls back order record creation. Cart remains intact.
- **E1.3:** Use case terminates in Failure state.

## Common Mistakes & CBAP Gotchas

> **⚠️ AI AGENT INSTRUCTION:** If any of these traps are detected in the user's request, surface a `⚠️ CBAP Gotcha` alert before generating the output.

**Trap 1 — Describing UI Layout Details inside Use Case Flows**
- Wrong: *"The customer clicks on the green button in the top right-hand corner of the page next to the logo."*
- Right: *"The customer submits the checkout form."* (Keep use cases technology-independent where possible so that UI redesigns do not invalidate the logic).

**Trap 2 — Designing God Use Cases**
- Wrong: Combining user registration, catalog search, item selection, payment, and support requests into a single, massive Use Case.
- Right: Break complex systems down into separate, atomic Use Cases linked by `<include>` or `<extend>` relationships.
