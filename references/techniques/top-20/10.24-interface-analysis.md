# Interface Analysis (BABOK 10.24)

## 1. Overview & Classical-to-Modern Bridge

Interface Analysis systematically identifies and describes the interaction boundaries between solutions, systems, hardware devices, or business roles. In modern enterprise architecture (dominated by ERPs, CRMs, and custom microservices), this technique is utilized to draft **System Integration Specifications**. 

To maintain high IT-proof rigor, the AI Agent must translate classical BABOK V3 elements directly to modern integration standards:

| BABOK V3 Element | Integration Equivalent | Practical Application |
| :--- | :--- | :--- |
| **1. Preparing for Analysis** | **Integration Topology Mapping** | Identify participating systems, direction of flow, and which system is the Source of Truth. |
| **2. Interface Identification** | **Interface Catalog & Protocol Selection** | Identify all required integration interfaces and select the transport protocol (REST API, Webhook, SFTP Batch). |
| **3. Interface Specification** | **Data Payload Mapping & Validation Specification** | Detail the exact JSON payloads, schema types, query parameters, CSV layouts, and error handling behaviors. |

## 2. System Integration Architecture: Upstream vs. Downstream

No enterprise system operates in isolation. When specifying system interactions, the AI Agent **MUST** establish the directional relationship and ownership of data entities:

*   **Source of Truth (SoT):** The single, authoritative system that owns a specific data entity (e.g. NetSuite ERP is the SoT for Invoice and General Ledger records; CRM is the SoT for Sales Opportunity records).
*   **Upstream System:** The system where data originates or is first created (e.g., SchoolsBuddy is Upstream for parent club registrations).
*   **Downstream System:** The system that consumes, subscribes to, or imports the upstream data (e.g., NetSuite ERP is Downstream for registrations to generate billing invoices).
*   **The Circular Sync Prohibition:** The AI Agent must never design bi-directional synchronization of the same fields without a master-slave reconciliation rule. A field must have exactly **one** Upstream Source of Truth.

## 3. The Unique ID & Migration Anchor Rule

> [!CAUTION]
> The single greatest failure point in enterprise system migrations (e.g. migrating from Legacy MIS A to New MIS B) and downstream integrations is the lack of a **Unique ID Mapping Registry**.

When analyzing integrations or migration pathways, the AI Agent **MUST** enforce the **Unique ID Anchor Rule**:
1.  **Define the Anchoring Key:** Identify an immutable, unique business key (e.g., Student ID, Tax Code, Email) or a system-generated UUID/GUID that remains constant across all systems.
2.  **Map the Integration Key Registry:** Every integration data payload must explicitly include:
    *   `active_id`: The current unique key in the new Source of Truth (MIS B).
    *   `legacy_id` *(if applicable)*: The historic key from the retired system (MIS A), vital for historical data reconciliation and downstream continuity.
    *   `correlation_id` / `external_id`: The corresponding unique identifier in the downstream system (e.g. NetSuite Customer Internal ID).
3.  **Explicit User Confirmation:** The AI Agent **MUST NOT** proceed with drafting integration schemas until the user has explicitly confirmed the Unique ID strategy.

## 4. Modern API Integration Specifications

Real-time integrations must be specified using clean, modern API design principles:

### A. RESTful Web Services (Real-time Synchronous)
*   **Method (HTTP Verbs):** `GET` (retrieve), `POST` (create), `PUT` (overwrite), `PATCH` (partial update), `DELETE` (remove).
*   **Endpoints:** Base URI + resource path (e.g., `POST https://api.school.edu/v1/payments`).
*   **Parameters:** Path variables (`/invoices/{invoice_id}`), query parameters (`?status=PAID&limit=50`), and headers (`Authorization: Bearer <Token>`).
*   **HTTP Response Status Codes:**
    *   `200 OK` / `201 Created`: Successful operations.
    *   `400 Bad Request`: Validation failure.
    *   `401 Unauthorized` / `403 Forbidden`: Authentication or authorization issues.
    *   `409 Conflict`: Duplicate record or unique constraint violation.
    *   `500 Internal Server Error`: Target system backend failure.

### B. Webhooks (Asynchronous Event-Driven)
*   Used when the Upstream system needs to push real-time event updates to a Downstream system as soon as they occur (e.g. Payment Gateway pushes `charge.successful` payload to ERP).
*   Must specify the **Callback URL** in the downstream system and the **Signature Verification Header** (security signature) to prevent spoofing.

## 5. SFTP Batch File Exchange Specifications (SFTP Fallback)

For legacy systems, ERPs with batch loading limitations, or high-volume daily reconciliations, a secure batch file exchange pattern is established:

*   **SFTP Connection Profile:** Transport via Secure FTP (Port 22) using SSH Key Exchange (RSA 4096-bit).
*   **Directory Layout Architecture:**
    *   `/inbound/`: Destination folder for files being uploaded by the external system.
    *   `/processing/`: Folder where files are moved while being actively parsed by the host system.
    *   `/archive/`: Storage folder for successfully imported files.
    *   `/error/`: Storage folder for files that failed schema validation or processing.
*   **File Layout Standards:**
    *   *Format:* Flat-file CSV (Comma Separated) or TSV (Tab Separated).
    *   *Encoding:* UTF-8 (strictly enforced).
    *   *Line Termination:* LF (`\n`) or CRLF (`\r\n`).
    *   *Naming Convention:* `[EntityName]_[Frequency]_[YYYYMMDD]_[HHMMSS].csv` (e.g. `STUDENT_DAILY_20260522_235959.csv`).
*   **Data Integrity Checksum:** Every file transfer must be accompanied by an **MD5 or SHA256 checksum** file (e.g. `STUDENT_DAILY_20260522_235959.csv.md5`) containing the hash of the file to verify complete transmission and prevent truncation errors.

## 🚦 System Integration Verification Checklist

- [ ] **SoT Definition:** Clear Source of Truth is defined for every data entity; circular synchronizations are completely avoided.
- [ ] **Unique ID Mapping:** The schema maps the old Legacy ID, the new Active ID, and the downstream External/Correlation ID.
- [ ] **Draw.io Compatibility:** Payload representations use clean, well-formed JSON syntax.
- [ ] **Directional Integrity:** Communication direction (CRM -> ERP, Webhook Callback -> App Backend) is explicitly defined for every interface.
- [ ] **Exception Paths & Retries:** Every API/file interface specifies error responses and defines retry limits/alert behaviors for communication timeouts.
- [ ] **File Validation (for SFTP):** Filename formats, delimiters, directory paths, UTF-8 encoding, and MD5 checksum checks are fully defined.
