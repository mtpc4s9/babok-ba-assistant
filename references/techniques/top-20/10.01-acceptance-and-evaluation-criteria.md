# Acceptance and Evaluation Criteria (BABOK 10.01)

## Overview

Defines measurable value-attribute criteria for objectively assessing solutions.
- **Acceptance Criteria** → pass/fail gate for a single solution (often contractual).
- **Evaluation Criteria** → weighted scoring scale to rank multiple options or designs.

## When to Use

- User needs to write UAT sign-off conditions for a single solution.
- User needs to compare and rank multiple design options or vendors.
- User needs to validate that a delivered solution meets agreed requirements.
- Task 7.5 (Define Design Options), 7.6 (Analyze Potential Value), 8.1 (Measure Solution Performance), 8.3 (Assess Solution Limitations).

## Do not use when
- When elicitation is still in progress and no solution candidates exist yet.
- When criteria cannot be defined because the solution space is not yet understood.

## Quick Reference

| Decision Point | Use This |
| :--- | :--- |
| Single solution, pass/fail gate | **Acceptance Criteria** |
| Multiple options, rank by value | **Evaluation Criteria** |
| Contractual sign-off required | **Acceptance Criteria** (flag change-control risk) |
| Stakeholder consensus on ranking needed | **Evaluation Criteria** (facilitate weighting workshop first) |
| Primary KAs | KA05, KA06, KA07, KA08 |
| Most common tasks | 7.5 (Evaluation), 8.1 (Acceptance) |

## Step-by-Step

> **🤖 AI AGENT INSTRUCTION:** If the context already makes the type obvious (e.g., Task 7.5 always implies multiple options → Evaluation Criteria), skip Step 1 and proceed directly to Step 2.

1. **Determine type** — If unclear, ask: *"Are we evaluating a single solution for sign-off, or comparing multiple design options?"*
2. **Identify value attributes** — Extract characteristics that substantially influence stakeholder value (e.g., performance, usability, security, scalability, reliability).
3. **Ensure testability** — Every criterion must be objectively measurable. If a criterion cannot be proven true/false or measured on a scale, prompt the user to clarify before proceeding.
4. **Generate output** — Fill the appropriate template below. Do not output both tables; select only the one that matches the context.

## Common Mistakes & CBAP Gotchas

> **⚠️ AI AGENT INSTRUCTION:** If any of these traps are detected in the user's request, surface a `⚠️ CBAP Gotcha` alert before generating the output.

**Trap 1 — Confusing Acceptance with Evaluation**
- Wrong: Using pass/fail for a multi-vendor selection. Using a ranking scale for a single solution sign-off.
- Right: Acceptance = one solution, pass/fail. Evaluation = multiple options, weighted scale.

**Trap 2 — Unmeasurable Criteria**
- Wrong: *"The system should be fast and easy to use."*
- Right: *"Page load time ≤ 2 seconds for 95th percentile of users under 500 concurrent sessions."*

**Trap 3 — Ignoring Contractual Rigidity of Acceptance Criteria**
- Wrong: Drafting acceptance criteria without noting they may form contractual obligations.
- Right: Flag to the user that acceptance criteria can be legally difficult to change. Establish a formal change control process before sign-off.

**Trap 4 — Single-Stakeholder Evaluation Weights**
- Wrong: Assigning weights to evaluation criteria based on one stakeholder's preference.
- Right: Facilitate a structured consensus session (e.g., workshop or survey) before finalizing weights. Misaligned weights invalidate the entire ranking.
