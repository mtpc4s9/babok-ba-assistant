# Document Analysis (BABOK 10.18)

## Overview

Document Analysis is an elicitation technique used to gather requirements, business rules, and constraints by systematically reviewing and analyzing existing materials. These documents can include system design specifications, corporate policy handbooks, regulatory standards, user guides, or competitor reports. It helps establish an operational baseline ("As-Is" state) without consuming stakeholder workshop time.

## When to Use

- User needs to research legacy system architecture before migration or decommissioning.
- User needs to align a solution with a new compliance mandate, government regulation, or corporate policy.
- Task 3.2 (Plan Stakeholder Engagement), Task 4.2 (Conduct Elicitation), Task 6.1 (Analyze Current State), Task 7.2 (Verify Requirements).

## Do not use when
- Documenting future-state desires, innovations, or target state capabilities that do not exist yet in any written form (use User Stories, Use Cases, or Business Model Canvas instead).
- Validating the prioritization of requirements (use Prioritization 10.33 instead).

## Step-by-Step for AI Agent

1. **Verify Context Inputs:** Scan for the raw text blocks, imported manuals, compliance regulations, or legacy specifications provided by the user.
2. **⚠️ SHARP NEGATIVE CONSTRAINT:** If the source document is completely missing or is a blank placeholder, STOP. Request the user to upload or paste the text of the source materials. An AI cannot perform document analysis on non-existent text.
3. **Parse and Filter Content:** Read the source material, filtering out fluff, and focus strictly on:
   - Functional capabilities ("What the system does")
   - Business rules & constraints ("What must or must not occur")
   - System qualities (NFRs: Performance, Security)
4. **Link to source:** Every extracted point must be linked to a specific section, page, or paragraph in the source document to ensure strict traceability.
5. **Generate Output:** Populate the standard template below.

## 🤖 AI AGENT INSTRUCTION (EXECUTION RULES)
- **Source Integrity:** Never paraphrase business rules or legal requirements in a way that alters their strict meaning. Use exact quotes where necessary, followed by parenthetical citations (e.g., *Sec 2.4, page 12*).
- **Gap Detection:** Explicitly flag outdated, contradictory, or ambiguous statements found within the analyzed documents.

## Common Mistakes & CBAP Gotchas

> **⚠️ AI AGENT INSTRUCTION:** If any of these traps are detected in the user's request, surface a `⚠️ CBAP Gotcha` alert before generating the output.

**Trap 1 — Treating Outdated Documents as Current Ground Truth**
- Wrong: Analyzing a design specification document dated 2018 and assuming it represents the current operational reality without validating against system logs.
- Right: Explicitly warn the user of the publication date risk. State that the document analysis represents the *written spec*, which might differ from the *live implementation*.

**Trap 2 — Failure to Cite Sources**
- Wrong: Providing a list of requirements without any reference back to the original document, leaving developers unable to trace the business origin.
- Right: Always mandate a strict, granular "Reference" column.
