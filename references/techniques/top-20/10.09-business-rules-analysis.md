# Business Rules Analysis (BABOK 10.09)

## Overview

Business Rules Analysis involves identifying, validating, and structuring rules that shape daily operational behaviors and system constraints. Unlike requirements, business rules are declarative statements of policy that exist independently of any software solution. They are typically structured to be atomic, testable, and represented in a clear format (e.g., "If-Then" constraints or mathematical bounds).

## When to Use

- User needs to document complex calculation formulas, validation logic, or authorization matrices.
- User needs to isolate policy logic from software code to enable rapid business rule changes.
- Task 7.1 (Specify and Model Requirements), Task 7.2 (Verify Requirements), Task 7.4 (Define Requirements Architecture).

## Do not use when
- Describing the sequential process steps or workflows of a system (use Process Modelling instead).
- Defining physical data storage schemas (use Data Dictionary instead).

## Quick Reference

| Rule Type | Definition | Example |
| :--- | :--- | :--- |
| **Definitional (Structural)** | Defines business terms or structural limits | "A Preferred Customer is one who has purchased > $10,000 in 12 months." |
| **Behavioral (Action-governing)** | Constrains or controls behavior, actions, or decisions | "If the account balance is < $0, the system must block outgoing transfers." |
| **Primary KAs** | KA03, KA04, KA05, KA07, KA08 | |
| **Most common tasks** | Task 7.1 (Specify & Model), Task 7.2 (Verify), Task 7.4 (Architecture) | |

## Step-by-Step for AI Agent

1. **Verify Context Inputs:** Scan the input documents (corporate policy documents, operational guidelines, or legacy systems) for rules.
2. **⚠️ SHARP NEGATIVE CONSTRAINT:** If the source document describes a sequence of steps (e.g., "Step 1: User logs in, Step 2: System shows dashboard"), do not write these as business rules. Re-route the user to Process Modelling (10.35). Business rules must be declarative statements, not procedural flows.
3. **Parse and Atomize Rules:** Break complex compound rules down into single, atomic rules (e.g., convert "Premium customers get free shipping and a 10% discount" into Rule A: free shipping, Rule B: 10% discount).
4. **Perform a Contradiction Check:** Ensure that no rules contradict each other (e.g., Rule A blocks transfers under $0, but Rule B permits overdrafts). Highlight any conflicts.
5. **Generate Output:** Populate the standard template below.

## 🤖 AI AGENT INSTRUCTION (EXECUTION RULES)
- **Declarative Form:** Always formulate rules in a declarative, state-enforcing style rather than step-by-step instructions. Use direct, clear verbs (e.g., "must", "must not", "is", "is not").
- **Atomicity:** Every rule in the catalog must represent exactly one logical constraint. If a rule contains "AND" or "OR" that alters outcomes, split it into separate line items.

## Common Mistakes & CBAP Gotchas

> **⚠️ AI AGENT INSTRUCTION:** If any of these traps are detected in the user's request, surface a `⚠️ CBAP Gotcha` alert before generating the output.

**Trap 1 — Writing Rules as System Implementation Requirements**
- Wrong: *"The database table must have an index on the user_id field."* (This is a design requirement, not a business rule).
- Right: *"Each user must be identified by a unique system-generated user ID."* (This is a structural business rule).

**Trap 2 — Confusing Rule Execution with the Rule Itself**
- Wrong: *"A supervisor logs into the console to approve the transaction."* (This is a process step).
- Right: *"Transactions exceeding $10,000 require authorization by a Supervisor role."* (This is a behavioral business rule).
