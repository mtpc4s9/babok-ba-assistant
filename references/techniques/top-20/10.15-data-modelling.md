#### description: Use when the user needs to describe the entities, classes, or data objects relevant to a domain, their attributes, and the relationships among them to provide a common set of semantics.

### Data Modelling (BABOK 10.15)

#### Overview
Defines a common vocabulary and structure for business information. It visually represents the elements that are important to the business, the attributes associated with those elements, and the significant relationships among them [1].
*   **Conceptual Data Model**  → Independent of any solution or technology; establishes a consistent business vocabulary [2].
*   **Logical Data Model**  → Abstraction that incorporates rules of normalization to manage data integrity [2].
*   **Physical Data Model**  → Describes how a database is physically organized (performance, concurrency, security) [2].

#### When to Use
*  User needs to define and communicate a consistent vocabulary used by domain SMEs and implementation SMEs [11].
*  User needs to formally manage the integrity of data and relationships for solution design [2].
*  User needs to identify missing requirements by exposing inconsistencies in how information is held by the business [12].
*  **Primary Tasks:** 4.2 (Conduct Elicitation), 5.2 (Maintain Requirements), 7.1 (Specify and Model Requirements), 7.4 (Define Requirements Architecture) [14].

#### Do not use when
*  User needs to show the sequential flow of work or activities (Use **Process Modelling** instead).
*  User needs to show the life cycle and status changes of a single entity (Use **State Modelling** instead) [15].

#### Quick Reference
| Context / Objective | Notation Standard | Key Terminology |
| ------ | ------ | ------ |
| Relational Database / General Data | **Entity-Relationship Diagram (ERD)** | Entity, Relationship, Cardinality [3, 8]. |
| Object-Oriented Software Design | **Class Diagram (UML®)** | Class, Association, Multiplicity, Operations [3, 8]. |

#### Step-by-Step
**🤖 AI AGENT INSTRUCTION:** When a user requests a data model, you MUST NOT generate arbitrary tables. Follow these exact steps to ensure BABOK V3 alignment.

1.  **Determine the Level of Abstraction** — Ask the user: *"Are we building a Conceptual model (business vocabulary), a Logical model (normalized for solution design), or a Physical model (database architecture)?"* [2, 3].
2.  **Identify Entities or Classes** — Extract the physical things, organizational units, abstract concepts, or events relevant to the domain [4]. 
3.  **Define Attributes** — For each entity/class, define the associated pieces of information, including Name, Values/Meanings, and Description [5, 6].
4.  **Establish Relationships & Cardinality/Multiplicity** — Define how entities connect. Use strict formats: *Each occurrence of [Entity A] is related to (minimum, maximum) of [Entity B]* [7, 8].
5.  **Generate Output** — Use the template below to present the structured Data Model Specification.
