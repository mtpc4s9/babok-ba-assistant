# Process Modelling (BABOK 10.35)

## 1. Overview & The Tri-Standard Framework

Process Modelling visually represents the sequential flow of work within an organization or system. To ensure high-quality communication with different audiences and guarantee successful IT verification, the AI Assistant must distinguish and utilize three distinct modeling standards:

| Standard | Target Audience | Core Purpose | Key Strengths |
| :--- | :--- | :--- | :--- |
| **1. Standard Flowchart** | Business stakeholders, C-level, general users | Present high-level business logic and sequence simply. | High readability, low technical barrier, easy to understand. |
| **2. BPMN 2.0** | Process owners, operations teams, BA teams | Map rigorous operational boundaries and exception handling. | Clear ownership (Pools/Lanes), standardized event & gateway logic. |
| **3. UML Activity Diagram**| Solution architects, system engineers, IT team | Define software execution logic, data flows, and system integration. | Precise control flows, object flows (data state changes), system actions. |

## 2. Notation & Symbol Specifications

### A. Standard Flowcharts (Lucidchart & BABOK 10.35.1)
*   **Terminator (Start/End):** Oval/Rounded Capsule. Indicates start or end points.
*   **Process:** Rectangle. Represents a single activity or action step.
*   **Decision:** Diamond. Represents a branching evaluation (Yes/No).
*   **Document:** Rectangle with a wavy base. Indicates report or document generation.
*   **Data (I/O):** Parallelogram. Denotes input data received or output data produced.
*   **Sub-Process (Predefined Process):** Rectangle with double vertical borders. Indicates a modular subprocess modeled in detail elsewhere.
*   **Database:** Cylinder. Represents data storage.
*   **Delay:** Capsule with a curved right side and flat left side. Denotes waiting or delay.

### B. BPMN 2.0 (Business Process Model and Notation)
*   **Pools & Swimlanes:** Pools represent independent participants (e.g., separate organizations). Lanes partition a Pool to represent internal roles/departments.
*   **Events (Circles):**
    *   *Start Event:* Thin-bordered circle. Initiates the process.
    *   *Timer Event:* Circle containing a clock. Indicates a time-based trigger.
    *   *Message Event:* Circle containing an envelope (catching/throwing messages).
    *   *Error Event:* Circle containing a lightning bolt (handles exceptions).
    *   *End Event:* Thick-bordered circle. Represents the final state.
*   **Activities/Tasks (Rounded Rectangles):**
    *   *User Task:* Performed manually by a human using a software tool.
    *   *Service Task:* Automated service execution without human intervention.
    *   *Sub-process:* Collapsed process represented by a `+` symbol at the bottom center.
*   **Gateways (Diamonds):**
    *   *Exclusive (XOR):* Blank or containing an `X`. Only one outgoing branch is executed.
    *   *Parallel (AND):* Containing a `+`. All outgoing branches execute concurrently.
    *   *Inclusive (OR):* Containing an `O`. One or more branches execute based on conditions.
*   **Connectors:**
    *   *Sequence Flow:* Solid line with solid arrowhead (cannot cross Pool boundaries).
    *   *Message Flow:* Dashed line with open circle start and open arrowhead (crosses Pools).

### C. UML Activity Diagrams
*   **Initial Node:** Solid black circle. Start of the activity.
*   **Activity Final Node:** Solid black circle enclosed in a ring (bullseye). Ends all flows.
*   **Flow Final Node:** Circle with an `X`. Ends a single specific path without ending the diagram.
*   **Action:** Rounded rectangle. Represents an atomic step in the software execution.
*   **Control Flow:** Solid arrow. Represents the execution sequence.
*   **Object Flow:** Solid/Dashed arrow passing through an **Object Node** (Rectangle) or a Pin (small attached square), representing the transfer of data or objects.
*   **Decision & Merge Nodes:** Diamonds. Decision splits a flow; Merge merges flows.
*   **Fork & Join (Synchronization Bars):** Thick horizontal or vertical bars:
    *   *Fork:* One incoming flow splits into multiple concurrent flows.
    *   *Join:* Multiple concurrent flows wait to synchronize into one outgoing flow.
*   **Partition (Swimlanes):** Vertical or horizontal columns separating software/system layers.

## 3. 🎯 Rule of Process Decomposition (Anti-Spiderweb Rule)

> [!WARNING]
> **DO NOT** create massive, monolithic diagrams that look like a "spiderweb". An unreadable process model is a failed deliverable.

To ensure clarity and maintainability, the AI Assistant must enforce strict **Process Decomposition**:
1.  **The Rule of 7±2:** A single process diagram must contain between **5 and 9 activity nodes** (Process steps, Tasks, or Actions).
2.  **Sub-Process Encapsulation:** If an operational sequence exceeds 9 nodes, group logical sub-steps into a single **Sub-Process** node (Predefined Process in Flowcharts or Collapsed Sub-process in BPMN).
3.  **Clean Interfaces:** Every decomposed Sub-Process must have a clearly defined **Trigger (Input)** and **End State (Output)** that matches its parent diagram parent-node interfaces.

## 4. 🔌 Draw.io & Mermaid Compatibility Engine

To guarantee that all generated Mermaid code imports **flawlessly** into Draw.io (via *Insert -> Advanced -> Mermaid*) and other standard visual tools without parser errors, the AI Assistant must adhere to these strict syntax guidelines:

1.  **Mandatory Label Double-Quoting:** Every node text containing spaces, parentheses, numbers, colons, or Vietnamese accents **MUST** be enclosed in double quotes.
    *   *Wrong:* `A[Start: Order Received]` or `B{Does file (PDF) exist?}`
    *   *Right:* `A["Start: Order Received"]` or `B{"Does file (PDF) exist?"}`
2.  **No HTML styling in Nodes:** Avoid using HTML tags (like `<br>`, `<b>`, `<i>`) inside node shapes. Draw.io's parser often breaks on tags. Use clean text formatting.
3.  **Strict Shape Mapping:**
    *   *Terminator / Start-End:* Capsule shape `([ ... ])`
    *   *Process / Task / Action:* Rounded rectangle `( ... )` or plain rectangle `[ ... ]`
    *   *Decision / Gateway:* Diamond shape `{" ... "}`
    *   *Sub-process:* Double-walled rectangle `[[ ... ]]`
    *   *Database / Data Store:* Cylinder `[( ... )]`
    *   *Data Input/Output:* Parallelogram `[\ ... /]` or `[/ ... /]`
4.  **Standard Connectors Only:** Use only `-->` for sequence flow and `-.->` for dashed/association flows. Do not use advanced formatting or complex arrow styles that are not natively supported by the older Mermaid parser version inside Draw.io.

## 5. Step-by-Step AI Execution Guide

When requested to draft a process model, the AI Agent must execute these steps:

1.  **Analyze Target Audience & Select Standard:**
    *   If business-focused / client-facing -> Use **Standard Flowchart**.
    *   If operational / human-system interaction -> Use **BPMN 2.0**.
    *   If software/IT systems integration -> Use **UML Activity Diagram**.
2.  **Apply the Anti-Spiderweb Rule:** Count the total estimated steps. If > 9 steps, divide the process into a Level 1 High-level map and Level 2 detailed Sub-processes.
3.  **Map the Happy Path:** Start with the trigger (Initial/Start node) and trace the quickest, error-free path to success.
4.  **Establish Exception Pathways:** Add decision gateways, write explicit guard conditions, and map alternate failure/retry paths.
5.  **Validate against Draw.io Compatibility rules:** Ensure all shapes match the standards, and all text labels are double-quoted.

## 🚦 IT Verification Checklist

> [!IMPORTANT]
> The AI Agent must verify the process model against this checklist before outputting it. If any check fails, reconstruct the flow.

- [ ] **Terminator Balance:** The process has exactly one clear start trigger and at least one defined end outcome.
- [ ] **Gateway Congruence:** Every decision gateway has 100% mutually exclusive branching paths (e.g. Yes vs No, Approved vs Rejected) and all outgoing paths are explicitly labeled.
- [ ] **Gateway Balance:** Exclusive or inclusive splits eventually merge back into a single path (using a merge node/gateway) or reach their respective end-state terminators. No orphan paths.
- [ ] **Swimlane Consistency:** Every activity node is strictly placed inside a Swimlane, Pool, or Partition representing a clear owner (Role or System).
- [ ] **Decomposition Validation:** No diagram exceeds 9 activity nodes. Sub-processes are used to hide complexity.
- [ ] **Mermaid-Draw.io Compatibility:** All labels containing spaces/special characters are double-quoted; no HTML tags inside shapes.
